▎異常現象

App 送出 epd_update 刷圖流程後，圖片資料有進入 convertToEpdBuffer() 並完成轉換，但接著在 NtxSpiUtility.resetEpd() 階段失敗，回傳 -1。上層最後收到 ntxSpi resetEpd() fail，並回報 action:epd_update FAIL。

▎最終判斷

這不是單純圖片轉檔、UI 或 Activity callback 問題。從 log 來看，真正的失敗鏈為：

  • Signage SDK 收到 epd_update intent，開始刷圖流程。
  • convertToEpdBuffer() 將 2560 × 720 圖片轉成 EPD buffer，流程成功返回。
  • 流程進入 resetEpd()，並執行 EPD / G2195 power sequence。
  • ntxSpi 透過 SPI 讀取 EPD Driver IC 狀態，但 8 顆 Driver IC 多次讀到 0x00 0x00 0x00 或無效資料。
  • Retry 100 次後仍無法取得所有 Driver IC response，回報 fail to get all drive IC status。
  • resetEpd() 回傳 -1，導致 epd_update 失敗。

▎重現頻率觀察

同一份 log 中可見多次 NtxSpiUtility.resetEpd() = -1，且多次伴隨 fail to get all drive IC status。這表示異常不是單次偶發 UI timeout，而是 EPD reset / Driver IC communication path 持續性失敗。

4 Root Cause 分析

▎直接原因：resetEpd 讀取 Driver IC 狀態失敗

Log 中連續出現 Driver IC [x] did not reply、retry 100 times、fail to get all drive IC status。因此 resetEpd() = -1 的直接原因，是 EPD Driver IC status check 沒有在 timeout 前通過。

▎可能根因 1：EPD panel / Driver IC 未正常上電

  • 雖然 AT+G2195PWR=1 回覆成功，但這只代表控制命令成功，不等於實際 VGH / VGL / VCOM / source / gate rail 都正常。
  • 若 G2195 enable 後實際輸出不穩或 delay 不足，Driver IC 仍可能完全不回應。
  • 建議以示波器或 DMM 量測 EPD power rails，而不是只看 AT command response。

▎可能根因 2：EPD reset pin / power sequence 時序異常

Log 中可見 setEpdPin ... on: 0 後進入電源與 Driver IC 檢查流程。若 reset pin 未正確 release、GPIO polarity 錯、delay 不足，或 power/reset 順序不符合 panel spec，都會造成 Driver IC 無法進入可通訊狀態。

▎可能根因 3：SPI read path 或 panel FPC 接觸異常

多數 Driver IC 回讀為 0x00 0x00 0x00，代表 MISO 沒有有效資料，可能是 SPI MISO / CLK / CS 線路、pinmux、driver node、FPC 接觸、panel 端連接或硬體焊接問題。若偶爾讀到類似 0x00 0xA2 0x89 但仍判定 did not reply，也需要確認讀值格式與 expected signature 是否一致。

▎可能根因 4：EPD 排線 / FPC 接觸不良

本次 log 呈現多顆 Driver IC 連續 did not reply，且常見 raw data 為 0x00 0x00 0x00。這種「所有 Driver IC 都無法正常回應」的型態，除了電源與 reset 時序外，也很符合 EPD 排線 / FPC 接觸不良造成 panel 端未連通或未醒來的情境。

  • FPC 未插到底、卡扣未壓緊、金手指氧化 / 髒污 / 偏位，都可能造成 Driver IC 沒有正常上電或通訊中斷。
  • 若 RESET / BUSY / SPI MISO / CLK / CS 任一關鍵訊號經由排線接觸不良，resetEpd() 就可能讀不到 Driver IC status。
  • 建議將排線接觸列為高優先排查項，先做重新插拔、清潔金手指、壓接確認、輕壓測試與 panel / 主板交叉測試。

▎檢查 3：確認 SPI read path 與 panel FPC

# 硬體側建議：
# 1. 重新插拔 EPD FPC，確認插到底且卡扣壓緊
# 2. 清潔 FPC 金手指與 connector，確認無氧化、髒污、刮傷、偏位
# 3. 輕壓 FPC / connector 重測 resetEpd，觀察是否由 fail 轉 pass
# 4. 量測 SPI CLK / CS / MISO / RESET / BUSY 是否有波形
# 5. 以正常 panel + 異常主板、異常 panel + 正常主板做交叉測試
# 6. 比對正常機與異常機 Driver IC readback

▎檢查 4：重現時只抓 EPD 關鍵 log

adb logcat -c
adb logcat -b all | grep -i -E "epd_update|NtxSpiUtility|resetEpd|convertToEpdBuffer|Driver IC|drive IC|EPDPWR|G2195PWR|setEpdPin|spiReceive"
7 建議修正方向
  • 先用正常機與異常機比對 resetEpd() 前後的 power sequence、GPIO 狀態與 Driver IC readback。
  • 確認 G2195 enable 後實際電壓是否到位，並確認電壓穩定時間是否足夠。
  • 確認 EPD reset pin 是否有正確 assert / release，且 delay 符合 panel / driver IC spec。
  • 檢查 SPI MISO / CLK / CS 以及 panel FPC 接觸；若全部 Driver IC 都不回，優先懷疑共用電源、reset、SPI bus 或 FPC 接觸不良。
  • 針對 EPD FPC 進行重新壓接、清潔金手指、connector 外觀檢查與輕壓測試；若重插後恢復，則排線接觸不良可列為主要根因。
  • 在 resetEpd() Driver IC status check 中補上 expected signature、raw data、retry index、GPIO / power 狀態，以便區分「完全無資料」與「讀到資料但格式不符」。
  • 若確認只是 power ready 時間不足，可嘗試增加 G2195 enable 後 delay 或 Driver IC check retry interval，但這只能作為 workaround，仍需確認硬體時序。
8 最終結論

目前 log 已可確認：EPD 刷圖不是卡在圖片轉換，而是卡在 resetEpd() 的 Driver IC 狀態確認階段。直接錯誤是 EPD Driver IC 無法在 retry 100 次內回應，導致 fail to get all drive IC status，最後 NtxSpiUtility.resetEpd() = -1，上層回報 epd_update FAIL。

優先處理順序建議為：

  • 先量測 EPD / G2195 實際電源輸出與 reset waveform。
  • 再確認 SPI read path、pinmux、FPC 接觸狀態與 Driver IC readback。
  • 若重新壓接 / 更換 panel 後恢復，則可將 EPD 排線 / FPC 接觸不良列為主要根因。
  • 最後才看上層刷圖流程或 callback 行為。
